# Development Workflow

Процесс разработки для изменений в `@april/tokens`, `@april/ui` и `@april/showcase`.

## 1. Типы изменений

- Token change: цвета, плотность, CSS-переменные, типографика.
- UI change: компоненты, провайдеры, тема Mantine, экспорт API.
- Showcase change: демонстрационные секции и сценарии проверки.

## 2. Рекомендуемая последовательность

1. Зафиксировать задачу в `tasks/` (цель, объем, критерии готовности).
2. Проверить влияние на публичный API (`packages/ui/src/index.ts`, `packages/tokens/src/index.ts`).
3. Внести изменения в исходники и сразу обновить showcase, если поведение визуальное.
4. Прогнать проверки:
   - `pnpm lint`
   - `pnpm typecheck`
   - `pnpm build`
   - `pnpm test`
5. Обновить документацию (`DESIGN_SYSTEM.md` и/или файлы в `docs/`), если поменялись правила/контракты. Релиз пакетов в registry — см. [`docs/PUBLISHING.md`](./PUBLISHING.md).

## 3. Definition of Done

- Изменения отражены в коде и showcase.
- Не сломан экспорт публичных пакетов.
- Линт, типы, сборка и тесты проходят.
- Обновлена документация по затронутому разделу.
- Добавлен отчет в задачу (если работа велась через `tasks/<id>/`).

## 4. Риски и анти-паттерны

- Не импортировать `UIKit` в production-сценарии потребителей пакета.
- Не менять токены без отражения в `DESIGN_SYSTEM.md`.
- Не добавлять зависимости в `@april/ui` без явной причины и оценки размера бандла.
- Не публиковать breaking changes без явной фиксации миграционного шага.
