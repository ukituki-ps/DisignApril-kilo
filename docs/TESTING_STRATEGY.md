# Тестовая стратегия DisignApril (v1)

Документ фиксирует подход к тестированию дизайн-системы April в репозитории `DisignApril`: какие уровни тестов обязательны, как они запускаются локально и в CI, и что считается quality gate для merge.

При подготовке стратегии использованы рабочие паттерны из `april-worker` и `april-profile-1`:
- единый источник истины по обязательным проверкам;
- явные pass/fail критерии и воспроизводимые команды;
- разделение на mandatory-слой и planned-расширения.

## 1. Принципы

- **Один источник истины по тестам:** этот документ + команды в `package.json`.
- **Пирамида тестирования:** больше unit, меньше integration, узкий слой e2e/smoke только для критичных пользовательских потоков.
- **Поведение важнее реализации:** UI-тесты проверяют публичное поведение компонента (DOM/a11y/события), а не внутренние state-переменные.
- **Стабильность без flaky:** избегать тестов, завязанных на тайминги и неявные async-побочные эффекты.
- **A11y как обязательная часть качества:** интерактивные элементы должны быть доступны по роли/лейблам и клавиатуре.

## 2. Область покрытия

Репозиторий `DisignApril` тестирует:
- `@april/ui` (компоненты, хуки, провайдеры, публичный API);
- `@april/tokens` (генерация/экспорт токенов, CSS-переменные, инварианты тем);
- `@april/showcase` (smoke-проверки ключевых сценариев витрины, без дублирования компонентных unit-тестов).

## 3. Уровни тестов

### 3.1 Unit (обязательный слой для компонентов)

Базовый стек:
- `Vitest` как раннер;
- `React Testing Library` для пользовательских сценариев;
- `jsdom` как среда браузера;
- `@testing-library/jest-dom` для DOM-утверждений.

Минимум для каждого нового или существенно измененного компонента:
- рендер базового состояния;
- ключевые интеракции (клик, ввод, переключение);
- callback-контракты (вызовы `on*` пропсов);
- важные ветки отображения (loading/empty/error/disabled, если применимо);
- базовые a11y-ожидания (доступность по роли/aria-label).

### 3.2 Integration (по необходимости)

Интеграционные тесты используются точечно, когда нужно проверить взаимодействие нескольких частей:
- `provider + component`;
- тема/плотность + компонент;
- публичные сценарии в `@april/ui`, где важна связка нескольких модулей.

Integration-слой не подменяет unit и не должен быть доминирующим по количеству тестов.

### 3.3 Showcase smoke (planned)

Для `apps/showcase` фиксируется узкий smoke-набор:
- приложение поднимается;
- основные демо-секции доступны;
- критичный сценарий нового компонента проходит в браузерном контуре.

Рекомендуемый инструмент для этапа расширения: `Playwright`.

## 4. Mandatory quality gate (цель v1)

Для merge в целевую ветку минимальный набор проверок:

1. `pnpm lint`
2. `pnpm typecheck`
3. `pnpm build`
4. `pnpm test` (после внедрения инфраструктуры тестов, см. задачи в `tasks/`)

Правило: любой `failed` или `cancelled` статус из обязательного набора блокирует merge до исправления причины и повторного зелёного прогона.

## 5. Локальный pre-merge прогон

```bash
pnpm lint
pnpm typecheck
pnpm build
pnpm test
```

Рекомендуемый порядок: сначала быстрые статические проверки, затем тесты.

## 6. Pass/fail критерии

- **Pass:** команда завершилась с exit code `0`.
- **Fail:** любой ненулевой exit code, падение assertions, runtime-ошибки раннера.
- Для flaky-падений обязательно повторное воспроизведение локально и фиксация первопричины в отчете задачи.

## 7. Матрица приоритетов покрытия

Приоритет `P0` (обязательно):
- компоненты публичного API (`packages/ui/src/index.ts`);
- компоненты с состояниями и пользовательским вводом;
- компоненты с несколькими режимами отображения (`inline/overlay`, `compact/comfortable`, `light/dark`).

Приоритет `P1` (после P0):
- сложные составные демо-сценарии `showcase`;
- edge-cases по адаптивности и длинным спискам;
- smoke браузерного контура.

## 8. Planned расширения (P1/P2)

- Добавить browser-smoke тесты для `apps/showcase` (Playwright).
- Добавить проверку snapshot/visual regression только для стабильных узлов и только при понятной операционной ценности.
- Добавить простую метрику качества по покрытию измененных файлов (без блокировки на первом этапе).
