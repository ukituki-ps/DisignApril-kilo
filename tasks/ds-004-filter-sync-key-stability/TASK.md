## Цель
Убрать нестабильность синхронизации filter-state в `CardListColumn`, чтобы изменение только порядка ключей в `filterValue` не воспринималось как реальное изменение данных.

## Контекст
- Репозиторий: `DisignApril`
- Затрагиваемые области: `packages/ui/src/components/CardListColumn.tsx`, тесты `packages/ui/src/components`
- Важные ориентиры:
  - компонент и его контракт фильтрации: `tasks/ds-001-card-list-column/TASK.md`
  - текущие тесты компонента: `tasks/ds-003-card-list-column-tests/TASK.md`
  - инженерные критерии: `docs/COMPONENT_STANDARDS.md`

## Входит в объем
- Проанализировать текущий механизм синхронизации `draftFilter` с внешним `filterValue`.
- Заменить ключ синхронизации на устойчивый к порядку полей подход:
  1. канонизация объекта фильтра (stable stringify с сортировкой ключей), или
  2. явное сравнение фактически используемых полей фильтра.
- Сохранить текущее поведение UX: пока модалка фильтров открыта, внешние изменения не должны затирать `draftFilter`.
- Добавить/обновить тесты на сценарий эквивалентных фильтров с разным порядком ключей (без ложной ресинхронизации).
- Проверить, что существующее поведение фильтра и callback-контракт (`onFilterChange`) не регрессирует.

## Не входит в объем
- Рефакторинг всей логики сортировки/поиска, не связанный с ключом синхронизации фильтра.
- Изменение публичного API компонента `CardListColumn`.
- Внедрение тяжелых deep-equality зависимостей без необходимости.

## Ограничения
- Следовать архитектуре монорепозитория на pnpm.
- Не ухудшить производительность на типичном размере `filterValue`.
- Решение должно быть предсказуемым для произвольного `Record<string, string | undefined>`.
