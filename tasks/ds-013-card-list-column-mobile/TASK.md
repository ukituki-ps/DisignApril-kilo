## Цель
Дать **`CardListColumn`** в `@ukituki-ps/april-ui` предсказуемое **мобильное** поведение на viewport **&lt;768px** (порог `sm` Mantine, как в §8): **один** основной режим отображения контента (карточки **в одну колонку**), перенос **панели управления** (поиск, фильтр, сортировка, добавление, переключение вида — по согласованному подмножеству) в **`AprilMobileShellBar`** и согласование **модальностей** с mobile-паттерном DS (нижний лист + действия в панели; **один активный контекст** в панели — §8).

**Зачем:** на телефоне текущая колонка с тулбаром в шапке и циклом трёх видов перегружает узкую ширину; единая зона большого пальца и норма «панель = один контекст» уже зафиксированы в DS — компонент должен **встраиваться** в этот контракт без дублирования логики в каждом микрофронте.

## Контекст
- Репозиторий: `DisignApril`; пакет: `packages/ui`; витрина: `apps/showcase` (**Mobile lab** — `apps/showcase/src/MobileShowcase.tsx`).
- Текущее состояние: `CardListColumn` и модули `packages/ui/src/components/cardListColumn/*` — **десктопно-ориентированный** тулбар и три вида (`list` | `grid` | `collapsed`) без ветки по breakpoint; в Mobile lab колонка **не** демонстрируется.
- Уже есть: `AprilMobileShellBar`, `AprilMobileBottomSheet`, `AprilVaulBottomSheet`, `aprilMobileShellBarContentPaddingBottom()`, норма §8 «один активный контекст в панели»; DS-011 зафиксировал виды и API вида/выбора.

## Архитектура (обязательные решения в PR)
- **Публичный API:** по возможности сохранить **один** импорт `CardListColumn`; новые пропсы — опциональные (например `mobileLayout`, `renderMobileToolbar`, `forceSingleColumnOnNarrow` — **имена выбрать в PR**), без ломающих изменений для существующих потребителей.
- **Определение «mobile»:** согласовать с §8 — **`useMediaQuery` / `(max-width: 47.99em)`** или проп **`isMobile`** от оболочки продукта; зафиксировать один подход в коде и в `DESIGN_SYSTEM.md` §13.
- **Виды на mobile:** зафиксировать в PR одно правило, например: только **сетка карточек с одной колонкой** (логически близко к `grid` с `minmax(0, 1fr)`), **без** цикла `list`↔`grid`↔`collapsed` **или** скрытый цикл только между согласованными подрежимами — без противоречия с «один режим» из постановки.
- **Нижняя панель:** либо компонент **принимает** слоты/колбэки для `AprilMobileShellBar` и сам не монтирует fixed-bar (композиция), либо **опционально** рендерит обёртку — выбрать один вариант, чтобы не плодить второй глобальный fixed слой в продукте.
- **Модалки:** фильтр / сортировка / add — на mobile по §11 связка **`AprilVaulBottomSheet`** (или упрощение по сценарию) + действия в **`AprilMobileShellBar`**; при открытом листе панель показывает **только** действия слоя (§8).
- **Документация:** обновить `DESIGN_SYSTEM.md` §13 (подпункт mobile Card List Column) и при необходимости §8 одной отсылкой.

## Входит в объем

### 1. Поведение контента на узкой ширине
- Режим **одной карточки в ряд** (адаптивная сетка или эквивалент), читаемые отступы и touch-ориентиры §8.
- Сохранить **выбор элемента**, **load more** (если есть), **скролл** контента; нижний **inset** контента при видимой `AprilMobileShellBar` — через `aprilMobileShellBarContentPaddingBottom()` (или документированная обёртка).

### 2. Панель управления
- Перенос действий поиска / фильтра / сортировки / add (и при необходимости **одной** согласованной кнопки вида) в паттерн **`AprilMobileShellBar`** (`leading` / `center` / `withSearch`), без нарушения правила **одного активного контекста** при открытых листах/формах.

### 3. Модальности
- Сценарии модалок колонки на mobile приведены к паттерну DS (лист + панель; порядок действий как в шапке `AprilModal`); обновить **встроенные примеры** в `CardListColumnSection` / витрине и **`CardListColumn.smoke.test.tsx`** под новые сценарии (селекторы, `aria`).

### 4. Витрина и тесты
- Секция в **Mobile lab** с `CardListColumn` в типичной ширине (~390 logical px или текущий кадр витрины), демонстрация панели + списка + открытия фильтра/формы.
- Обновить smoke-тесты; `pnpm lint`, `pnpm typecheck`, `pnpm build`, `pnpm test`.

## Не входит в объем
- Полная мобильная адаптация **Kanban** / **React Flow** (отдельные эпики).
- Виртуализация длинных списков.
- Новый публичный компонент **только** для мобильной колонки, если это противоречит цели «один импорт» (согласовать в PR — предпочтение фасада `CardListColumn`).

## Ограничения
- Монорепозиторий на **pnpm**; не импортировать **`UIKit`** в production-код пакета `ui`.
- Иконки — `AprilIcon*` / `lucide-react` по §11; не вводить второй набор.
- Не хардкодить произвольные hex; плотность — `useDensity` там, где меняются размеры контролов.
- Не смешивать в **`AprilMobileShellBar`** глобальные табы приложения с кнопками модалки колонки на одном экране-демо без явного сценария «только контекст колонки».
