- Приоритет: high

- Связанные документы: `DESIGN_SYSTEM.md`, `docs/COMPONENT_STANDARDS.md`, `docs/AGENT_TASK_TEMPLATE.md`

## Цель
Реализовать базовый компонент списка карточек в стиле колонки Kanban, который можно использовать как самостоятельный строительный блок интерфейса (Tasks/CRM/HRM). Компонент должен закрывать базовые сценарии работы со списком: поиск, организация элементов и переключение формата отображения.

## Контекст
- Репозиторий: `DisignApril`
- Затрагиваемые области: `packages/ui`, `apps/showcase`, `docs`
- Важные ориентиры:
  - паттерн Kanban: `DESIGN_SYSTEM.md`
  - требования к API и качеству: `docs/COMPONENT_STANDARDS.md`

## Входит в объем
- Создать компонент списка карточек (Kanban-like column) с двумя вертикальными секциями:
  1. Верхняя секция управления:
     - поиск через input (обязательно);
     - фильтрация (опционально, допускается через modal);
     - сортировка (опционально, допускается через modal);
     - добавление нового элемента (опционально, допускается через modal).
  2. Нижняя секция списка карточек.
- Добавить два формата отображения компонента:
  - **Inline**: список встроен в интерфейс; должна быть кнопка переключения в компактный вид (аналогично поведению sidebar).
  - **Overlay**: по кнопке список открывается поверх интерфейса; повторное нажатие закрывает.
- Обеспечить единый размер карточек:
  - карточки могут иметь разный контент/структуру;
  - визуальный размер карточки в рамках одного списка всегда одинаковый.
- Реализовать изменение ширины списка через перетягивание правой границы:
  - доступный диапазон ширины: от `15%` до `50%` от контейнера.
- Добавить showcase-сценарий с демонстрацией:
  - базовый список;
  - переключение inline/overlay;
  - изменение ширины;
  - поиск по карточкам.

## Не входит в объем
- Полноценный drag-and-drop между несколькими колонками.
- Серверная загрузка/сохранение данных.
- Сложные комбинации фильтров и продвинутый query-builder.
- Виртуализация длинных списков.

## Ограничения
- Следовать архитектуре монорепозитория на pnpm.
- Не хардкодить цвета/отступы: использовать токены и тему.
- Соблюдать a11y для интерактивных элементов:
  - кнопки и иконки с `aria-label`;
  - управление клавиатурой для основных действий открытия/закрытия и переключения режима.

## Требования к API (минимум)
- Компонент принимает:
  - массив карточек;
  - конфигурацию режима (`inline` / `overlay`);
  - настройки ширины (`min=15%`, `max=50%`, `default`);
  - callbacks на поиск/добавление/изменение режима.
- API должен быть минимальным и предсказуемым (без лишних пропсов).
