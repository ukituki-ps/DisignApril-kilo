## Цель
Реализовать в `@ukituki-ps/april-ui` **нормативный компонент нижней панели управления** для viewport **&lt;768px** по договорённостям April: **одна** плавающая капсула внизу экрана со слотами **слева (часто «Назад») | центр (табы и/или контекстные действия) | справа (поиск с раскрытием поля)**; учёт **safe-area**, основа для **нижнего отступа** у скроллящегося контента.

Добить **отдельную витрину mobile** в `apps/showcase`: не полагаться только на сужение окна браузера вокруг `UIKit`, а показывать примеры **внутри фиксированных контейнеров** типичных размеров телефона (логические px), чтобы верстка и переполнение проверялись предсказуемо.

**Зачем:** единый паттерн shell для микрофронтов, меньше расхождений с §8 DS, эталон для a11y (фокус поиска, Escape, имена иконок).

## Контекст
- Репозиторий: `DisignApril`; пакеты: `packages/tokens` (опционально), `packages/ui`, витрина: `apps/showcase`.
- Сейчас showcase: один `AppShell` + `<UIKit />` в `apps/showcase/src/App.tsx` — широкая область, mobile-паттерн из §8 **не продемонстрирован** отдельно.
- Иконки: `AprilIcon*` / `lucide-react` по `DESIGN_SYSTEM.md` §11; модалки форм — **`AprilModal`** (DS-010), в задаче достаточно **одного** демо «панель + модалка» на mobile-витрине без переписывания всего `UIKit`.
- Роутера в showcase **нет** — переключение «десктопная витрина / mobile-lab» делать **локальным состоянием** (например `useState` + `Tabs`/`SegmentedControl` в шапке) или эквивалент без новых зависимостей.

## Архитектура (обязательные решения в PR)
- **Имя компонента:** выбрать одно и зафиксировать в коде и в `DESIGN_SYSTEM.md` §11 (например `AprilMobileShellBar` или `AprilMobileBottomBar`) — не оставлять два конкурирующих имени в публичном API.
- **Слоты:** публичный API через явные пропсы (`leading`, `center`, `trailing` **или** `back`, `tabs`, `actions`, `search` — выбрать одну схему) + при необходимости **render props** для гибкости; **поиск** — контролируемое/полуконтролируемое раскрытие с **переносом фокуса** в `TextInput` при открытии и закрытием по **Escape** (см. §10).
- **Фиксированное позиционирование:** панель у нижнего края viewport приложения; `padding-bottom` с учётом `env(safe-area-inset-bottom)` на корне панели.
- **Inset для контента:** экспортировать **константу высоты** и/или хук **`useMobileShellBarInset()`** (или согласованное имя), возвращающий число px для `padding-bottom` у `main`, чтобы продукты не дублировали магические числа (документировать в §11 и в JSDoc).

## Входит в объем

### 1. Токены (по необходимости, минимально)
- Если без новых токенов нельзя согласовать touch-зазоры между командами — добавить в `packages/tokens` константы (например минимальный tap target **44**, зазор между иконками в панели) и при необходимости отражение в `tokens.css`; иначе **достаточно** задокументированных констант в `packages/ui` рядом с компонентом с пометкой в PR «токены follow-up».

### 2. Компонент в `packages/ui`
- Реализация на **Mantine 7** (`Paper` / `Group` / `ActionIcon` / `TextInput` и т.д.), стили **teal / нейтрали April** через theme vars, без произвольных hex вне темы.
- Состояния для витрины и типов:
  - **корень:** центр с табами (например Mantine `SegmentedControl` или кнопки-табы) + 1–2 контекстных действия + поиск;
  - **drill-down:** слева «Назад», центр — действия, справа — поиск; **глобальные табы скрыты** (как в §8);
  - **поиск раскрыт:** центр временно отдаётся строке поиска (или задокументированный альтернативный UX — зафиксировать один вариант в коде витрины).
- **a11y:** `aria-label` у icon-only; для табов в центре — осмысленная разметка и клавиатура (см. §10); не ломать порядок табуляции при раскрытии поиска.
- Экспорт из `packages/ui/src/index.ts`; **smoke-тест** (например `AprilMobileShellBar.smoke.test.tsx` или рядом с компонентом): монтирование слотов, открытие/закрытие поиска (по возможности без хрупких снапшотов).

### 3. Отдельная mobile-витрина в `apps/showcase` (обязательно)
- **Не** ограничиваться узкой колонкой внутри существующего `UIKit` как единственным способом смотреть mobile.
- Добавить отдельный режим/страницу (**Mobile lab** / **Витрина mobile**): переключатель в шапке `App.tsx` между текущим **`UIKit`** и новым корневым компонентом (например `MobileShowcase.tsx` в `apps/showcase/src/`).
- Внутри mobile-витрины — **несколько контейнеров «аппаратного» размера** (логические px), минимум два пресета, например:
  - **390×844** (типичный iPhone ширина × высота viewport-ориентир);
  - **360×800** (типичный Android).
- Каждый контейнер: **фиксированная ширина** (и заданная высота или `min-height` с внутренним скроллом), визуально отделён от фона страницы (**рамка**, **скругление**, опционально лёгкая **тень**), подпись с размерами; **внутри** — мини-приложение: условный `AppShell` или `Stack` с **скроллящимся контентом** (длинный список-заглушка) с **`padding-bottom` = inset от панели** + демо **нижней панели** внизу контейнера (как в реальном встраивании в iframe/viewport телефона).
- Опционально для демо safe-area: задать на обёртке контейнера тестовые `padding-bottom` / CSS-переменные, имитирующие inset, **или** коротко описать в README витрины, что в Safari на устройстве проверяется отдельно.

## Не входит в объем
- Рефакторинг **всех** секций `UIKit` под mobile и перенос всей галереи в узкие контейнеры.
- Реализация **полного** продуктового shell (роутинг, auth, уведомления) в showcase.
- Адаптация **Kanban**, **React Flow**, **`CardListColumn`** под mobile-layout (отдельные эпики; в этой задаче достаточно **упоминания** в демо или одной статичной заглушки «контент»).
- Отдельное приложение Vite вне монорепозитория.
- Зависимость **react-router** только ради этой задачи.

## Ограничения
- Монорепозиторий на **pnpm**; не импортировать **`UIKit`** в production-код пакета `ui`.
- Не вводить второй набор UI-иконок; для кнопок панели — **`AprilIcon*`** / Lucide по §11.
- Согласовать с **Mantine breakpoints** (`sm` = 768): компонент и демо должны быть согласованы с порогом **&lt;768px** из §8 (комментарий в коде / использование `useMediaQuery` при необходимости только на витрине).
